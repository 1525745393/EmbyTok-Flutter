# 发布流程代码审查 - 产品需求文档 (PRD)

## Overview

- **Summary**: 对 EmbyTok 项目的 Flutter/Android 发布流程相关变更进行全面代码审查，确保代码质量、安全性、可维护性和性能符合项目标准。审查范围包括 5 个变更文件，935 行新增代码，涉及自动发布脚本、验证脚本、回滚脚本、CI/CD 工作流和 Makefile 构建目标。

- **Purpose**: 防止在发布流程中引入潜在 bug、安全问题或架构不一致，确保发布流程在各种边界条件下的稳定性。发布流程是项目的关键路径 — 任何发布流程的错误都可能导致版本混乱、用户下载错误、数据安全风险或 CI/CD 不可用。

- **Target Users**: 
  - 项目维护者（负责发布新版本）
  - CI/CD 工程师（负责维护工作流）
  - 移动端开发者（依赖 Flutter/Android 构建）

## Goals

1. **代码质量**: 确保新增代码符合项目的 Bash 脚本和 YAML 约定标准，命名清晰、逻辑简洁
2. **错误处理完整性**: 确保所有失败路径都有适当的处理和用户提示
3. **安全性**: 检查签名密钥管理、权限设置和敏感信息泄露风险
4. **性能**: 评估构建缓存策略和脚本执行效率
5. **可维护性**: 确保代码结构清晰、注释充分，便于未来维护
6. **测试覆盖**: 验证是否有适当的测试策略和回滚机制
7. **配置管理**: 检查硬编码值和默认值的合理性

## Non-Goals (Out of Scope)

1. **功能实现审查之外的代码修改**: 本审查任务只识别问题，不在此阶段直接修复代码（除了极简单的注释/格式问题）
2. **其他模块代码审查**: 仅限本次发布流程相关的 5 个变更文件，不审查 backend/ 下的其他代码或 frontend/lib/ 下的 Flutter UI 代码
3. **完整的安全审计**: 本审查聚焦于发布流程的安全性，不进行全项目安全渗透测试
4. **性能基准测试**: 不执行实际基准测试，只审查性能相关的代码逻辑
5. **自动测试用例编写**: 仅审查现有测试策略，不在此任务中新增测试

## Background & Context

**历史背景**: 
- 项目当前版本 v1.2.4，tag 已推送至 GitHub
- 发布流程此前依赖人工手动操作，缺乏自动化和验证机制
- 本次变更是对项目发布流程的首次系统化重构

**技术栈限制**:
- **Bash 脚本**: 需在 macOS/Linux/GitHub Actions runner 上均可执行，需兼容 Bash 3.2+（macOS 默认版本）
- **Flutter/Dart**: 版本为 3.24.0，使用 Gradle 进行 Android 构建
- **Android**: minSdk=21, targetSdk=34, compileSdk=34
- **GitHub Actions**: 使用 Ubuntu latest runner

**已知的项目特点**:
- 项目同时有前端（Flutter）和后端（Python），发布流程需要同时更新两个技术栈的版本号
- 缺少单元测试，主要依赖集成测试和手动验证
- CHANGELOG.md 采用 Keep a Changelog 格式
- 版本号遵循 Semantic Versioning 2.0

## Functional Requirements

### FR-1: 代码风格一致性审查

**描述**: 验证所有新增代码是否遵循项目代码风格约定，包括命名规范、注释格式、缩进等。

**具体检查项**:
- 变量和函数命名是否有意义、可读性强（避免 `a`、`b`、`tmp` 等模糊命名）
- Bash 脚本中的双引号使用是否一致（字符串扩展时的引号包裹）
- 脚本的头部注释（shebang、功能描述、使用示例）是否完整
- 缩进是否统一（2 或 4 空格）
- 代码组织是否合理，相关逻辑是否放在一起
- 函数是否只做一件事（单一职责原则）
- 中文注释是否简洁清晰，解释"为什么"而不是"做什么"

### FR-2: 错误处理完整性审查

**描述**: 验证所有可能失败的操作都有适当的错误处理机制。

**具体检查项**:
- `set -euo pipefail` 是否正确设置并理解其行为
- 关键操作（如 `git tag`、`git push`、`sed`、`grep`）失败时是否有错误提示
- 是否存在 `$?` 的不正确使用（应优先使用 `if cmd; then ...` 模式）
- 空值/空字符串的处理（如变量未设置时是否有默认值）
- 条件判断中的 `-z`、`-n`、`-f`、`-d` 等文件测试运算符是否正确
- 管道失败是否使用 `pipefail` 正确捕获
- 脚本退出码是否有意义（0=成功，非0=失败，且不同错误类型有不同退出码）

### FR-3: 安全性检查

**描述**: 验证代码不引入安全漏洞，正确处理敏感信息。

**具体检查项**:
- 签名密钥和密码是否通过环境变量（GitHub Secrets）而非硬编码方式处理
- 是否有日志输出中泄露敏感信息（如密码、密钥内容）
- 文件权限设置是否合理（keystore 文件不应可读）
- 是否检查命令存在性（如 `command -v` 或 `which`）
- 输入验证：用户输入是否做了基本格式校验
- 是否存在命令注入风险（变量在命令中的使用是否安全）
- `curl`/`wget` 等网络命令是否使用 `-sSf` 或类似安全选项（本项目中可能不涉及）

### FR-4: 性能影响评估

**描述**: 评估代码变更对构建和发布性能的影响。

**具体检查项**:
- CI 缓存策略是否有效（key 是否基于文件 hash，restore-keys 是否合理）
- 是否存在不必要的重复计算/重复命令
- 大文件操作是否有内存优化
- 并行优化是否合理（多个 job 并行 vs 串行）
- `find`/`grep` 命令是否有不必要的递归
- 构建产物清理策略是否合理
- Gradle daemon 和 Flutter 性能相关参数是否配置得当

### FR-5: 可维护性分析

**描述**: 评估代码的长期可维护性。

**具体检查项**:
- 是否有完整的文件头部注释描述脚本功能和使用方法
- 是否有行内注释解释复杂或不直观的逻辑
- 是否有统一的日志输出函数（如 `log_info`、`log_success`）
- 是否有魔法数字或硬编码字符串（应提取为常量或配置）
- 变量作用域是否合理（是否使用 `local` 关键字在函数内）
- 是否使用了 `readonly` 标记只读常量
- 是否有 TODO/FIXME 标记未解决的问题
- 脚本之间的逻辑是否有重复（应考虑提取公共函数库）

### FR-6: 测试覆盖情况

**描述**: 验证代码变更是否有配套的测试策略。

**具体检查项**:
- 是否有 dry-run 模式（预览操作而不实际变更）
- 是否有手动验证步骤的说明文档
- CI 工作流中是否包含发布前验证
- 失败场景是否有测试覆盖
- 边缘/边界条件是否有考虑（空输入、脏工作树、网络失败等）

## Non-Functional Requirements

### NFR-1: 可移植性

- 脚本应兼容 Bash 3.2+（macOS 自带版本）和 Bash 4+（Linux）
- 避免使用 GNU 特有选项（如 GNU `sed` `-i` 行为与 BSD `sed` 不同）
- `date` 命令格式应跨平台兼容（注意 macOS date 和 GNU date 差异）
- 不依赖外部工具（如 `jq`、`yq`）除非已在 CI 中预装

### NFR-2: 可用性/用户体验

- 脚本输出应清晰、结构分明，使用颜色增强可读性
- dry-run 模式应详细展示即将执行的操作
- 错误信息应具体、可操作（告诉用户怎么修复）
- 交互式确认（y/N）应安全，默认为 No

### NFR-3: 可靠性

- 脚本在非预期输入时应优雅降级而非崩溃
- 幂等性：多次运行相同命令不应导致不同结果（dry-run 是，实际 release 可能不是）
- 失败时应保留足够的诊断信息（日志、上下文）

### NFR-4: 文档完整性

- 每个脚本应有头部注释描述
- Makefile 目标应有简短描述
- CI workflow 的每个 step 应有清晰的 name

## Constraints

- **技术限制**: 只能使用 GitHub Actions runner 默认预装的工具（不能假设用户安装了额外工具）
- **兼容性**: 必须兼容 macOS（本地开发）和 Linux（CI 环境）
- **发布策略**: tag 推送后自动触发发布，不能阻塞主分支
- **安全约束**: 不能在构建日志中输出密码或密钥内容

## Assumptions

1. 项目用户具备基本的 Git 操作能力，理解 tag 和 release 的概念
2. Android 签名密钥已正确配置在 GitHub Secrets 中
3. 项目用户使用 Bash 或兼容的 shell
4. `pubspec.yaml` 版本号格式为 `MAJOR.MINOR.PATCH`（不包含 `-beta` 等预发布标识）
5. 发布操作由维护者在本地执行 `scripts/release.sh` 后自动触发 CI

## Acceptance Criteria

### AC-1: 代码风格一致性验证
- **Given**: 5 个变更文件的完整代码内容
- **When**: 人工审查代码风格和结构
- **Then**: 识别出所有命名不清晰、结构混乱、注释不足的代码片段
- **Verification**: `human-judgment` — 审查者对照项目已有代码风格进行判断
- **Notes**: 重点关注 Bash 脚本中重复出现的逻辑块和公共函数

### AC-2: 错误处理完整性验证
- **Given**: 脚本中的所有错误路径和异常处理
- **When**: 模拟各种失败场景（命令失败、网络失败、文件缺失）
- **Then**: 识别出未处理的错误路径和用户不友好的错误信息
- **Verification**: `human-judgment` + `programmatic`（可执行脚本进行 dry-run 验证）
- **Notes**: 重点检查 `set -euo pipefail` 的设置是否在所有关键脚本中一致

### AC-3: 安全性验证
- **Given**: 所有涉及敏感信息的代码段
- **When**: 审查密钥管理、日志输出、权限设置
- **Then**: 识别出所有安全隐患（硬编码密码、日志泄露敏感信息、不安全的文件权限）
- **Verification**: `human-judgment`
- **Notes**: 重点关注 `android-release.yml` 中的签名配置和环境变量处理

### AC-4: 性能影响验证
- **Given**: CI 缓存配置和脚本执行流程
- **When**: 分析缓存 key 设计、重复操作、资源使用
- **Then**: 识别性能问题并提出优化建议
- **Verification**: `human-judgment` + `programmatic`（可执行 CI 构建观察时间）
- **Notes**: 重点关注 Gradle 和 Flutter pub 缓存

### AC-5: 可维护性验证
- **Given**: 所有脚本的结构、注释和文档
- **When**: 审查代码结构的清晰度和注释完整性
- **Then**: 识别出难以维护的代码结构和缺失的文档
- **Verification**: `human-judgment`
- **Notes**: 重点评估脚本之间的重复代码量和可复用性

### AC-6: 测试和边界条件验证
- **Given**: 脚本的所有执行路径和输入验证
- **When**: 审查边界条件处理（空输入、脏工作树、网络失败等）
- **Then**: 识别出未覆盖的测试路径和边界条件处理不当的问题
- **Verification**: `human-judgment` + `programmatic`（手动执行边界测试）
- **Notes**: dry-run 模式和实际发布模式的一致性是关键

## Open Questions

- [ ] **问题 1**: `scripts/release.sh` 中使用 `sed -i` 进行文件修改，macOS 下 BSD sed 与 Linux sed 的 `-i` 选项行为不同，是否需要考虑这个兼容性问题？
  - **影响**: 如果维护者在 macOS 本地执行脚本，可能导致 sed 操作失败
  - **优先级**: 高

- [ ] **问题 2**: `scripts/release.sh` 第 258 行 `git add -A` 会提交所有未跟踪的变更，这在发布流程中是否安全？
  - **影响**: 可能意外提交不应包含在发布中的文件
  - **优先级**: 高

- [ ] **问题 3**: 三个脚本（release/verify/rollback）都有重复的颜色和日志函数（约 25-30 行），是否应该提取为公共的 `scripts/_common.sh` 或类似文件？
  - **影响**: 代码重复、维护成本高、修改一处需要同步所有脚本
  - **优先级**: 中

- [ ] **问题 4**: `.github/workflows/android-release.yml` 中 `aab_build_time` 的默认值处理（第 245 行 `${{ steps.build-aab.outputs.aab_build_time || 0 }}`）在 debug 模式下 AAB 未构建时，是否会导致后续步骤失败？
  - **影响**: debug 模式下 create-release job 可能因找不到 AAB artifact 失败
  - **优先级**: 中

- [ ] **问题 5**: `scripts/rollback-release.sh` 的"回滚"是否应包含代码回退（如 `git reset`）？目前仅删除 tag，不回退代码
  - **影响**: 用户可能误解"回滚"的含义
  - **优先级**: 中
