# 播放器控制功能增强 - 产品需求文档 (PRD)

## 概述
- **摘要**：本项目旨在增强 EmbyTok-Flutter 的视频播放器控制功能，参考上游项目 EmbyTok（React/TypeScript 版）的优秀设计，新增/优化以下功能：纯净模式（沉浸观看）、增强的 TV/遥控器模式、更丰富的播放速度选项、下一集/上一集快捷切换、完善的自动播放与提示、更智能的进度条显示策略。目标是提升用户体验，使移动端、平板和大屏幕 TV 用户都能获得更流畅直观的视频浏览与播放体验。
- **目的**：解决当前播放器在以下场景的痛点：(1) 横屏沉浸式观看时 UI 元素过多干扰视线；(2) TV/机顶盒环境下遥控器操作体验不佳；(3) 播放速度选择有限；(4) 短内容也显示进度条造成界面拥挤；(5) 无法快速在剧集间切换。
- **目标用户**：移动端竖屏浏览用户、横屏全屏观看用户、TV/盒子遥控器用户、平板用户、剧集追更用户（番剧/短剧/电视剧）。

## 目标
- **目标 1**：实现纯净模式（Pure Mode），用户可一键隐藏所有覆盖层 UI 元素，提供完全沉浸的观看体验。
- **目标 2**：增强 TV 模式适配，支持遥控器方向键导航，明确的焦点高亮，提供更适合大屏设备的控制界面。
- **目标 3**：扩展播放速度档位（0.5x/0.75x/1.0x/1.25x/1.5x/2.0x），长按保持 2x 快放功能保持并优化。
- **目标 4**：完善下一集/上一集（Next/Prev）切换功能，在播放控制条中提供明确的切换入口。
- **目标 5**：实现智能进度条策略 - 仅对大于 3 分钟的视频显示底部细进度条，短内容保持界面简洁。
- **目标 6**：优化自动播放（Auto Play）机制，提供清晰的自动播放提示与开关入口。
- **目标 7**：增加"随机浏览"模式，支持一次性加载 80 条随机内容，提升发现新内容的效率。

## 非目标（Scope Out）
- **不做**：重写视频播放核心库（video_player 保持不变，不引入新的视频播放引擎）。
- **不做**：在线字幕自动搜索与翻译（超出本次范围，仅优化已有字幕轨道切换）。
- **不做**：视频下载/离线播放功能。
- **不做**：视频推荐算法（仅提供随机列表，不做个性化推荐）。
- **不做**：支持 iOS 平台的特殊控制（如原生 AVKit 控制层替换）。

## 背景与上下文
**当前项目状态（EmbyTok-Flutter）**：
- 基于 Flutter + Riverpod 实现，使用 `video_player` 作为播放内核。
- 已实现功能：TikTok 风格上下滑动切换视频、底部控制条（播放/暂停/进度/倍速）、右侧操作栏（全屏/静音/点赞/收藏）、手势层（单击切换控制/双击点赞/长按倍速/水平拖动进度）、字幕控制面板、横屏全屏模式、播放降级链、播放状态上报到 Emby 服务器、预加载机制、观看历史续播。
- 相关文件：`lib/widgets/video_page_item.dart`、`lib/widgets/video_controls.dart`、`lib/widgets/gesture_overlay.dart`、`lib/widgets/video_player_widget.dart`、`lib/widgets/subtitle_controls.dart`、`lib/providers/video_playback_controller.dart`、`lib/utils/constants.dart`。

**参考项目功能（EmbyTok React 版）**：
- 技术栈：React 18 + TypeScript + Vite + Tailwind CSS + Capacitor + Lucide React。
- 关键功能亮点：无限播放（Infinite playback mode）、纯净模式（Pure Mode，隐藏所有 UI 元素）、TV 模式（大屏幕遥控器友好 UI）、长按 2 倍速播放、左右滑动快进快退、大于 3 分钟视频显示进度条、随机列表 80 条视频、自动播放提示、智能适配横竖屏视频、多级导航深入浏览剧集/季/集并逐步返回、续播同步播放位置。

**技术约束**：
- 必须保持与现有 Riverpod 状态管理架构一致，新状态通过新 Provider 或扩展现有 Provider 实现。
- 不引入新的第三方重度依赖（保持 Flutter 生态的标准组件）。
- 必须同时支持竖屏流式浏览（Feed View）和横屏全屏模式（Fullscreen）。
- 必须保持对 Emby 服务器 API 的兼容，支持 PlaybackStopped/PlaybackStart 等上报。

## 功能需求

### FR-1：纯净模式（Pure Mode）
- 用户在全屏播放时可切换"纯净模式"，启用后隐藏底部控制条、右侧操作栏、标题信息等所有覆盖层 UI 元素。
- 纯净模式下，单击屏幕重新呼出控制层（同现有单击切换控制逻辑扩展）。
- 纯净模式状态在设置中持久化存储（Shared Preferences），下次进入默认为用户上次的选择。
- 纯净模式在竖屏流式浏览中默认关闭，横屏全屏时提供快捷入口。

### FR-2：增强的播放速度选择
- 当前播放速度选项从 3 档（1x/1.5x/2x）扩展为 6 档（0.5x/0.75x/1.0x/1.25x/1.5x/2.0x）。
- 长按倍速功能保持不变（长按期间强制 2x），松开后恢复用户选择的基准倍速。
- 循环切换策略优化：单击倍速按钮时，弹出速度选择面板（而非直接循环切换），让用户快速选择。
- 用户选择的播放速度持久化存储。

### FR-3：下一集/上一集切换
- 在底部控制条中添加"上一集"、"下一集"按钮（仅在有前后剧集时可用）。
- 点击"下一集"：切换到当前媒体库中的下一条视频，保留播放器初始化状态。
- 支持滑动手势作为备选：双指左滑/右滑切换视频（替代方案）。
- 在网格视图中也应标记正在播放的条目，方便用户定位。

### FR-4：智能进度条显示策略
- 视频时长 < 3 分钟：不显示底部细进度条（保持界面简洁）。
- 视频时长 >= 3 分钟：显示底部细进度条（`video_page_item.dart` 中 `_ThinProgressBar`）。
- 拖动进度时：无论视频长短，均显示拖动预览条与时间戳（当前已实现）。
- 进入/退出时的动画保持流畅（200ms 渐入渐出）。

### FR-5：TV 模式与遥控器支持
- 检测设备类型/屏幕尺寸，提供 TV 模式 UI（更大的字号、更粗的焦点边框、间距增加）。
- 遥控器方向键支持：上下切换视频、左右拖动进度、OK 键播放/暂停、返回键退出全屏。
- 焦点可视化：当前聚焦元素高亮边框 + 轻微缩放动画。
- TV 模式下默认开启"纯净模式 + 自动播放"组合（可关闭）。
- （可选增强）：键位提示气泡，首次进入 TV 模式时展示 5 秒后自动消失。

### FR-6：自动播放优化与提示
- 新增 `isAutoPlay` Provider（当前项目可能已有）：控制视频结束后自动切换到下一条。
- 自动播放启用时，在控制条中显示"自动播放已开启"标记（小图标 + 文字提示）。
- 视频切换时提供渐入动画叠加，确保过渡不突兀。
- 自动播放在用户主动暂停时不触发，避免干扰。

### FR-7：随机列表模式
- 在顶部工具栏添加"随机一波"按钮，点击后从当前媒体库随机拉取 80 条视频刷新播放列表。
- 与"最新"、"收藏"模式并列的浏览模式选项。
- 随机模式切换时保留当前播放位置，允许用户后续"继续观看"。

### FR-8：播放状态上报与续播增强
- 确保所有新增播放控制操作（倍速变更、进度拖动、切换剧集）都触发正确的 Emby PlaybackProgress 上报。
- 支持续播恢复：在视频列表中，已观看过的条目展示上次播放位置标记。
- 续播点标记：底部细进度条上添加小圆圈标记上次停止位置（可选增强）。

## 非功能需求

### NFR-1：性能
- 所有 UI 状态变更响应延迟 < 100ms。
- 纯净模式启用/禁用切换动画时间控制在 300ms 内。
- 播放速度切换应在 1 个视频帧周期内生效，不触发重新初始化播放器。
- 视频切换期间内存占用保持稳定（避免重复创建 VideoPlayerController 造成的 OOM 风险，继续使用现有预加载策略）。

### NFR-2：可用性与可访问性
- 控制按钮最小触摸目标 >= 44x44 dp。
- 文字与背景对比度 >= 4.5:1（WCAG AA 标准）。
- 遥控器焦点元素有明显视觉反馈（边框 + 阴影 + 缩放）。
- TV 模式字体比标准模式大 20%。

### NFR-3：代码组织
- 新增/修改文件保持现有命名与导入规范。
- Provider 集中在 `lib/providers/` 目录。
- Widget 组件集中在 `lib/widgets/` 目录。
- 常量集中在 `lib/utils/constants.dart`。

### NFR-4：向后兼容
- 所有新增功能均提供"默认关闭"或"保持原行为"的开关。
- 不破坏现有 API 调用与数据结构。
- 升级用户保留原有观看历史与设置。

## 约束

### 技术约束
- Flutter SDK：保持与当前项目相同的版本约束（`pubspec.yaml` 中的 `sdk`）。
- 依赖库：`video_player`、`flutter_riverpod` 保持现有版本。
- 不引入新的平台专用插件（保持 Android/Web 兼容性）。

### 业务约束
- Emby 服务器 API 保持不变，不要求服务器端修改。
- Plex 支持在后续版本中考虑（本次仅关注播放器体验优化）。

### 依赖约束
- `video_page_item.dart` 的修改需特别注意：此处包含完整的播放上报链与控制层，改动需要仔细测试。
- `video_controls.dart` 是本次修改最多的组件。

## 假设
- 用户设备有足够的内存处理 Flutter 渲染的多层覆盖层（当前项目已有类似结构，风险低）。
- Emby / Jellyfin 服务器均支持 PlaybackProgress/PlaybackStart/PlaybackStopped 的标准上报 API。
- 大于 3 分钟的进度条显示策略基于经验值，后续可调整。

## 验收标准

### AC-1：纯净模式切换
- **给定**：用户进入横屏全屏播放状态
- **当**：用户点击右侧操作栏的"纯净模式"按钮（或长按屏幕空白区域 1.5 秒）
- **则**：所有覆盖 UI 元素（底部控制条、右侧操作栏、标题信息）以 300ms 渐隐动画消失；再次单击屏幕时，所有 UI 元素重新渐显
- **验证**：人工测试
- **备注**：纯净模式状态保存在本地存储中

### AC-2：播放速度选择面板
- **给定**：用户正在播放视频
- **当**：用户点击倍速按钮
- **则**：弹出半透明选择面板，展示 6 档速度（0.5x/0.75x/1.0x/1.25x/1.5x/2.0x），当前速度高亮；点击选择后面板消失并应用新速度
- **验证**：代码走查 + 人工测试

### AC-3：长按倍速与基础速度分离
- **给定**：用户当前选择基础速度为 1.25x
- **当**：用户长按屏幕 500ms 触发倍速，持续观看 10 秒后松开
- **则**：长按期间为 2.0x，松开后回到 1.25x；Emby 服务器收到的播放速度状态正确反映变更
- **验证**：网络请求抓包（人工测试）+ 代码走查

### AC-4：下一集/上一集按钮可用性
- **给定**：用户正在播放某媒体库中的第 3 条视频
- **当**：观察底部控制条
- **则**："上一集"按钮在首条时禁用（灰色），"下一集"按钮在末条时禁用；中间条目均可用
- **验证**：人工测试

### AC-5：智能进度条策略
- **给定**：播放两条测试视频，一条时长 1 分 30 秒，一条时长 10 分钟
- **当**：分别播放时观察底部
- **则**：短视频无底部细进度条；长视频有底部细进度条；两者拖动进度时均显示预览与时间戳
- **验证**：人工测试

### AC-6：TV 模式遥控器焦点导航
- **给定**：设备连接外部遥控器（或模拟器模拟）
- **当**：用户按下方向键（上/下/左/右/OK/返回）
- **则**：在播放界面正确响应：上/下切换视频、左/右拖动进度、OK 播放/暂停、返回退出全屏；当前焦点元素有明显的视觉高亮
- **验证**：人工测试

### AC-7：随机浏览模式拉取
- **给定**：用户已登录并选择某媒体库
- **当**：用户点击"随机一波"按钮
- **则**：从当前媒体库随机拉取 80 条视频刷新播放列表；动画过渡与进入最新模式时一致
- **验证**：人工测试

### AC-8：自动播放状态提示
- **给定**：用户开启自动播放
- **当**：视频播放中观察控制条右侧
- **则**：显示"自动播放"图标与状态提示（On/Off），可点击切换；切换后立即生效并持久化
- **验证**：人工测试

### AC-9：播放状态上报一致性
- **给定**：用户使用新的播放控制操作（倍速变更、进度拖动、切换剧集）
- **当**：操作完成后查看 Emby 服务器的播放状态
- **则**：服务器正确反映当前播放位置、播放状态、播放速度；用户下次进入"继续观看"列表能看到续播点
- **验证**：人工测试 + 网络请求检查

### AC-10：性能与稳定性
- **给定**：连续使用播放功能 30 分钟并频繁切换视频
- **当**：观察应用稳定性与内存
- **则**：无崩溃、无明显内存增长、无渲染卡顿（60fps 保持）
- **验证**：人工测试 + DevTools 性能分析
